🚩 Feature Flags — Architecture Guide

Platform-specific implementation across AWS, GCP, Internal Kubernetes, and On-Premise. Includes the definitive RF counting policy.

The RF Policy — Definitive Statement

✓ Counts as RF
A production deployment — code reaches the production environment through the standard pipeline, regardless of whether a feature flag is ON or OFF.

Example: Deploy payments-service v2.1 to prod with new payment flow hidden behind a flag = 1 release counted.
✗ Does NOT count as RF
A pipeline run that deploys to Dev/Test/Staging only.

A flag toggle (enabling a feature that was already deployed).

A config change that doesn't go through the standard pipeline.

Any deployment not captured in DataSight as a production deployment.

Why Feature Flags Increase RF (Indirectly)

Feature flags don't add RF points by themselves. They remove the psychological and technical barrier to releasing more frequently to production.

Code merged
→
CI passes
→
Deploy to PROD +1 RF
→
Flag OFF
Feature hidden
→
Flag ON when ready
(no new deploy needed)
The key insight: Without feature flags, teams wait until the whole feature is "done" before deploying — big batches, low RF. With feature flags, they deploy incomplete features safely to production every sprint, hidden behind a toggle. Each of those small deployments counts. RF increases naturally because releasing small is now safe.

Platform-Specific Implementation via Standard Pipeline

Each deployment platform requires a different technical approach. All approaches must be integrated into the org standard pipeline — no ad-hoc toggles.

☁️
AWS
Java/Spring · Node · Python on EC2/ECS/Lambda
Recommended: AWS AppConfig
AWS AppConfig is the native feature flag service. Integrates with your Jenkins pipeline via the AWS CLI. Flag values are fetched at runtime — no redeployment needed to toggle.
# In your standard Jenkins pipeline (Jenkinsfile): stage('Deploy to Prod') { steps { # Deploy the application code (THIS counts as RF) sh 'aws ecs update-service --cluster prod --service ${APP_NAME}' # Create/update feature flag config in AppConfig sh ''' aws appconfig deploy \ --application-id ${APPCONFIG_APP_ID} \ --environment-id prod \ --configuration-profile-id ${FLAG_PROFILE_ID} \ --deployment-strategy-id AppConfig.Linear50PercentEvery30Seconds ''' } }
In your app code, check the flag: if (appConfigClient.isEnabled("new-payment-flow")) { ... }
🌐
GCP
Java · Node · Python on GKE/Cloud Run/GCE
Recommended: Firebase Remote Config
Firebase Remote Config works for any GCP app — not just mobile. Flags are JSON values fetched at runtime. Alternatively, use GCP Secret Manager for simple boolean flags.
# In your Jenkinsfile for GCP apps: stage('Deploy to Prod') { steps { # Deploy (counts as RF) sh 'gcloud run deploy ${SERVICE_NAME} --image gcr.io/${PROJECT}/${IMAGE}' # Update feature flag in Remote Config via REST API sh ''' curl -X PUT \ "https://firebaseremoteconfig.googleapis.com/v1/projects/${GCP_PROJECT}/remoteConfig" \ -H "Authorization: Bearer $(gcloud auth print-access-token)" \ -H "Content-Type: application/json" \ -d '{"parameters": {"new_payment_flow": {"defaultValue": {"value": "false"}}}}' ''' } }
Simpler alternative: use GCP Secret Manager with a naming convention like ff_new_payment_flow = "false". Toggle via gcloud CLI without a redeployment.
⚙️
Internal Kubernetes (K8s)
Any stack — deployed on internal K8s platform
Recommended: ConfigMap-based flags
No external service needed. Use a dedicated ConfigMap for feature flags in each namespace. The pipeline deploys both the application AND updates the ConfigMap. App reads flags from the ConfigMap at startup or via a config reload endpoint.
# feature-flags.yaml — checked into your repo apiVersion: v1 kind: ConfigMap metadata: name: feature-flags namespace: payments-prod data: NEW_PAYMENT_FLOW: "false" # flip to "true" via kubectl or pipeline ENHANCED_SEARCH: "false" --- # In Jenkinsfile: stage('Deploy') { steps { # Deploy app image (counts as RF) sh 'kubectl set image deployment/${APP} app=${NEW_IMAGE} -n payments-prod' # Apply feature flags ConfigMap (declarative) sh 'kubectl apply -f k8s/feature-flags.yaml -n payments-prod' } }
Flag values must be in a SEPARATE ConfigMap from app config — never mix feature flags with DB passwords or secrets. Flag ConfigMaps should be in Git and versioned with the app.
🏢
On-Premise / Traditional
Java/Spring · .NET · Any legacy on-prem stack
Recommended: Database-backed or config-file flags
For apps that can't use cloud services. Two approaches: (1) A shared flags table in an existing database, or (2) A properties/config file deployed alongside the app and read at startup.
# Option A: Properties file (simplest) # feature-flags.properties — deployed with the app feature.new_payment_flow=false feature.enhanced_search=false # In Spring app: @Value("${feature.new_payment_flow:false}") private boolean newPaymentFlowEnabled; --- # Option B: DB table (better for runtime toggling) CREATE TABLE feature_flags ( flag_name VARCHAR(100) PRIMARY KEY, enabled BOOLEAN DEFAULT FALSE, updated_at TIMESTAMP ); # Toggle without redeployment: UPDATE feature_flags SET enabled=true WHERE flag_name='new_payment_flow';
For .NET apps: use Microsoft.FeatureManagement NuGet package — it reads flags from appsettings.json and supports runtime toggling via Azure App Configuration or a local JSON file.

Standard Pipeline Requirements for Feature Flags

Regardless of platform, ALL feature flag implementations must meet these standards to qualify for the feature_flags_adopted: true registry flag.

Requirement Rule Why
Declared in pipeline YAMLFlags must be defined in pipeline config — no ad hoc toggles outside the pipelineAuditability, reproducibility
Default to OFFNew flags must default to false/off in all environmentsSafety — deploy never accidentally enables an incomplete feature
Flag lifecycle trackedFlags older than 90 days without being enabled are "stale" and must be removed or renewedPrevents flag debt accumulating in codebase
Naming conventionff_[feature_name]_[jira_ticket] e.g. ff_new_checkout_PAY123Links feature to ticket, enables auto-cleanup when ticket closes
No direct prod toggleEnabling a flag in prod must go through a pipeline step or approved runbook — not a direct DB query or console clickChange management compliance
For iSeries / Vendor apps: Feature flags are not applicable. These apps have externally-controlled release schedules. The feature_flags_adopted flag should remain false for these apps and it does not affect their DPI score (RF is already excluded for vendor apps).
